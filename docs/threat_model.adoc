== Threat Model

Redpanda's quality goals name "Unlinkability" and "Private enough for high-risk requirements", but a privacy property is only testable against a defined adversary. This chapter defines the adversary classes Redpanda reasons about, states for each what the system defends against **today**, what is **planned** (and in which milestone), and what is explicitly **out of scope**. Being explicit about what is *not* covered is a deliberate goal: "private enough" is meaningless without it.

_Last reviewed: 2026-07-09, against milestone state MS01–MS06 + MS08 Done (see `milestones/00_status_overview.md`). The binding source for every mechanism named below is the Decisions section of the respective milestone master spec._

=== Adversary Classes

[options="header",cols="1,3"]
|===
| Class | Description
| **(A) Local network observer** | Sees the client's traffic on the local link / access network (Wi-Fi, ISP first hop). Knows the client's IP and the IP of the full node it connects to, plus packet sizes and timing.
| **(B) Single malicious full node / malicious OH host** | One node the client talks to is adversarial. Most relevant: the node hosting the client's own Outbound Handle, which the client must trust to store and serve its mail.
| **(C) Colluding subset of nodes** | Several nodes cooperate to correlate traffic across hops (e.g. a relay and the destination OH host comparing notes).
| **(D) Global passive adversary** | Observes the entire network simultaneously (a powerful ISP coalition or nation-state). The reference adversary against which "high-risk" privacy must ultimately be measured.
| **(E) Device seizure** | The adversary physically obtains an unlocked or compromised device, including its key material and local database.
|===

=== (A) Local Network Observer

* **Defended today:** Transport is framed AES-256-GCM over an X25519 key exchange with Ed25519 node identities (handshake v23, live since MS03 — see link:milestones/ms03_authenticated_encryption.md[MS03 Decisions]). Content and protocol fields are not readable on the wire, and the transport is authenticated: a tampering observer produces frames that fail GCM authentication and are rejected, not silently accepted. The observer learns that the client talks to a full node, not what is said.
* **Remaining gap — closed:** The legacy v22 light-client gate (unauthenticated AES-CTR) was switched off 2026-07-09 (sdd02 Phase 1, MS03 Decision 10) and the dormant code itself — the v22 handshake path, the emergency flag and the legacy ECDSA fallback in `OutboundAuth` — was **removed** in sdd02 Phase 2 (2026-07-19, redpandaj https://github.com/redPanda-project/redpandaj/pull/261[#261], MS03 Decision 11). The v22 downgrade surface no longer exists; rejected v22 handshakes are still counted (without IPs) for residual-traffic observability.
* **Out of scope:** The observer still sees *that* the client connects to a Redpanda node, when, and roughly how much it sends/receives. Traffic-shape obfuscation (cover traffic, padding to fixed cadence) is **not** planned — see (D). Hiding Redpanda usage itself (pluggable-transport-style) is a non-goal.

=== (B) Single Malicious Full Node / Malicious OH Host

The OH host is the most sensitive single-node relationship: the client registers a mailbox on it and fetches from it repeatedly. It must be named for what it is — **a semi-trusted, per-user component**, effectively "a small central server for that one user", even though the network as a whole is decentralized.

What the OH host learns about its client:

* the client's **IP address** (and thus rough location / network);
* the client's **fetch rhythm** — when and how often it polls or is awake;
* the **number, size, and timing** of incoming messages deposited in the mailbox;
* once push lands (MS07, spec still open), the **wake-up moment** of the device (a push is a timing signal even with no content).

* **Defended today:**
** *Content and sender identity* — the OH host does not learn message content (end-to-end Double Ratchet since MS03b, see (E)) nor the sender identity of deposited mail (the sender is whoever forwarded the packet). Mailbox content is opaque ciphertext.
** *Mailbox flush* — deposit is hardened since MS02b (reject-new eviction, per-item size limit, byte quota, register rate limit — see link:milestones/ms02b_oh_discovery_forwarding.md[MS02b Decisions]); a malicious party can no longer silently displace stored mail.
** *Client location vs. senders* — since MS05, reply traffic reaches the client's mailbox via sender-built onions over recipient-chosen hops (Reverse Garlic Blocks); a channel partner needs the client's `oh_id` but never its network location (see link:milestones/ms05_reverse_garlic.md#decisions-backend-ms05-2026-06-13[Backend-MS05 Decisions], Decision 6 — Bob knowing Alice's `oh_id` is a documented trade-off of the hop-descriptor RGB model).
** *Availability signals* — since MS06, routing ACKs and channel ACKs are separated (see link:milestones/ms06_two_layer_ack.md#decisions-backend-ms06-2026-07-03[Backend-MS06 Decisions]). The R-ACK is generated by the depositing node and deliberately carries no `message_id` — only a timestamp and a status code; the sender correlates it via a single-use `ack_session_tag` that is meaningless to relays and OH hosts. The Channel-ACK (delivery receipt) travels channel-encrypted like any other message, so the OH host cannot distinguish receipts from regular mail. Session tags in general are single-use random 16-byte values; the client discards unknown/consumed tags before decrypting and cleans up outstanding tags after 48 h (link:milestones/ms05_reverse_garlic.md#decisions-frontend-ms05-2026-07-02[Frontend-MS05 Decisions], Decision 6).
* **Planned:** MS07 (push) is the remaining item on this front — see the push section below. Note one accepted metadata trade-off until then: when a pending RGB is expired or missing, the client falls back to the untagged forward path via the recipient's `oh_id` (link:milestones/ms05_reverse_garlic.md#decisions-frontend-ms05-2026-07-02[Frontend-MS05 Decisions], Decision 3), so the OH host sees a mix of tagged and untagged deposits.
* **Out of scope:** The OH host inherently sees its client's IP, fetch rhythm, and message volume/timing. Redpanda does **not** hide these from the chosen OH host — mitigation is to choose/operate a trusted OH, or to use several. Splitting metadata across multiple OHs and self-hosting are the intended user-side defenses; neither is enforced by the protocol.

=== (C) Colluding Subset of Nodes

* **Defended today:** Multi-hop forward paths (MS04) and reverse-garlic reply paths (MS05) are live: no single relay sees both endpoints of a conversation — each relay learns only the next hop, relays stay stateless, and packets are fixed-size (2048 bytes) with a 5-minute `packet_id` dedup against byte-identical replays. Path diversity comes from fresh random hop selection per message and per RGB (with prefix diversity); forward and return hops are *not* forced to be disjoint — a deliberate KISS trade-off, impossible to guarantee in small networks (link:milestones/ms05_reverse_garlic.md#decisions-frontend-ms05-2026-07-02[Frontend-MS05 Decisions], Decision 5). An attacker must control *all* hops of a path to link sender and recipient.
* **SURB/Sphinx review (done):** The reply-block attack classes known from I2P SURBs / Sphinx were reviewed retroactively with Backend-MS05 (link:milestones/ms05_reverse_garlic.md#decisions-backend-ms05-2026-06-13[Backend-MS05 Decisions], Decisions 7–8): *tagging* — replies are normal sender-built onions with no malleable pre-encrypted material; every layer is GCM-authenticated with the next hop as AAD, so a marked packet dies at the next hop. *Replay* — relay-side `packet_id` dedup plus single-use session tags at the endpoint. *Size correlation* — fixed 2048-byte packets; the per-hop ciphertext-length shrink remains accepted per MS04 Decision 5 (mitigation would be dummy traffic — deferred).
* **Out of scope / open risk:** If colluding nodes control enough of the path — or if the node population is small (the unsolved full-node-economy problem, see `11_risks_and_technical_debt.adoc`) — correlation becomes feasible. With few real operators the network degenerates toward "the project's own nodes", which collapses this defense. Timing correlation across colluding hops without cover traffic remains possible (see (D)). Note also that the R-ACK is a routing *hint*, not proof: the final return hop or the OH host could fake or drop it, which is accepted as the trust model (MS06 master spec, Open Question 5) — cryptographic confirmation only comes from the Channel-ACK.

=== (D) Global Passive Adversary

* **Defended today:** Effectively nothing specific to this adversary. Three hops and 2048-byte padding do not stop an observer who sees the whole network: end-to-end **timing correlation** (a packet entering at the sender and a matching one arriving at the recipient's OH) is trivial without traffic shaping.
* **Planned:** No milestone currently targets this adversary. The building blocks that would be required — **cover traffic, send delays, and mix-style batching** (the Nym/Loopix approach) — are not specified anywhere in the roadmap.
* **Out of scope (explicit):** Redpanda is **out of scope against a global passive adversary** today and for the foreseeable roadmap. The "high-risk" claim must be read with this limit: Redpanda raises the cost of correlation for local, single-node, and limited-collusion adversaries, but does **not** claim resistance to a global observer without cover traffic / mix delays. This should be stated plainly to users rather than implied away by "3 hops + padding".

=== (E) Device Seizure

* **Defended today:** Channel encryption uses a Double Ratchet (Envelope v4) since MS03b (link:milestones/ms03b_forward_secrecy.md[MS03b Decisions]). After the first full round trip of a channel, message keys contain fresh DH entropy that the QR-carried bootstrap key `K_enc` alone cannot reproduce; old chain and message keys are not retained (best-effort — Dart cannot zeroize GC memory, but neither the session object nor the persisted state ever hold past keys, and unit tests verify that a state snapshot does not decrypt earlier messages and that a cloned state is locked out after one round trip). Consequently, a full node that stored ciphertext can no longer decrypt it retroactively with key material seized later.
* **What seizure still compromises today:**
** the **locally stored plaintext message history** — the app database stores decrypted messages; forward secrecy protects ciphertext in transit and at nodes, not the local history. OS-level device encryption is the only defense here;
** the **current ratchet state**, including skipped message keys (bounded: max 512 skipped per advance, max 1024 stored per channel, 30-day expiry — MS03b Decision 4). This decrypts messages inside that out-of-order window and, until the peer's next DH ratchet step, newly arriving messages;
** **`K_enc`**, which still lives in the channel QR and on the device as the ratchet bootstrap (MS03b Decision 3): a channel that never completed a full round trip is entirely derivable from it, and a screenshot/backup of the QR has the same effect for such channels.
* **Planned:** Removing `K_enc` from the QR requires per-device key exchange and remains deferred (MS03 Frontend-Decision 3 / MS03b Decision 3). No at-rest "panic"/deniable-storage feature is planned. Since MS03, the QR carries public keys + `K_enc` only — `k_auth_priv` no longer travels in it.
* **Out of scope:** Deniability is listed as a quality goal but conflicts with long-lived signing keys and is not delivered by any milestone.

=== The Push Notification Leak (FCM / APNs)

Push deserves its own treatment because it reintroduces a central observer the rest of the design works to avoid. Using Google FCM or Apple APNs to wake the device leaks "this user is receiving a message **now**" (a timing oracle) to Google/Apple, even though the content stays encrypted. For a project whose core promise is metadata protection, this cannot be silently accepted.

Concrete mitigation options — to be **decided in MS07**, not deferred indefinitely:

* **Self-hosted / decentralized push:** UnifiedPush with a self-hosted ntfy server removes Google/Apple from the loop (Android; iOS remains constrained by APNs).
* **Batching / random delay:** Coalesce or randomly delay push wake-ups so the push timing no longer maps one-to-one to message arrival, weakening the timing oracle.
* **Polling-only mode for high-risk users:** Allow disabling third-party push entirely and falling back to client-driven polling, trading battery for the elimination of the FCM/APNs leak.

The decision (which combination, and the default for the standard vs. high-risk profile) belongs in MS07's spec — still open as of this revision (MS07 status: Missing; a decision memo covering these options is in preparation). Documenting it as an open decision — rather than as a two-line accepted risk — is the point.

=== Supply Chain: The P2P Updater

This does not fit the adversary classes above — it is not about leaking metadata but about a single compromised secret leading directly to code execution across the network, so it is called out on its own.

Every full node asks its peers for a newer signed JAR/APK right after the handshake and, on a validly-signed newer one, installs and restarts into it (`InboundCommandProcessor.handleUpdateAnswerContent` et al., CMD bytes 9–16). This makes the updater signing key the highest-value secret in the project: whoever controls it can push arbitrary code to every full node that accepts the update, and there is no code-signing transparency log or second signer. The project deliberately operates this channel rather than removing it (updater removal-vs-operate decision, Option B, 2026-07-09) and mitigates the risk with an offline signing ceremony, a build-time rollback floor (`Updater.MIN_UPDATE_TIMESTAMP_MS`) against downgrade attacks, and fail-closed key handling when no valid key is configured. As of 2026-07-11 the network runs on an interim testnet key (redpandaj PR #244); rotating it to a properly-ceremonied production key is tracked as backlog task T13. There is no revocation mechanism if the key is compromised — see <<key-compromise,Updater Key Ceremony, Section 5>> for the full procedure and the accepted structural limitation.
