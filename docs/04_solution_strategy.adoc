== Solution Strategy

=== Core Metaphor: "Flaschenpost" (Message in a Bottle)
Unlike connection-oriented protocols (like XMPP or TCP tunnels), Redpanda treats messsages as independent packets ("Flaschenpost") that navigate the network hop-by-hop.
*   **Fire & Forget**: The sender throws the message into the network.
*   **Store & Forward**: If the receiver is not ready, the message waits at a Full Node ("drift").
*   **Asynchronous**: No requirement for both parties to be online simultaneously.

=== Fundamental Concepts

==== 1. Garlic Routing (The Forward Path)
Inspired by I2P, messages are bundled. A "Garlic" package can contain multiple "Cloves" (messages).
*   **Layered Encryption**: Each hop peels off one layer.
*   **Bundle Efficiency**: Multiple messages for same route segments are grouped to reduce traffic correlation.

==== 2. Reverse Garlic (The Return Path / Factory)
To avoid direct DHT lookups (which leak metadata), Redpanda uses **Reverse Garlic Reply Blocks**.
*   Alice creates a "Reply Block" (pre-encrypted path back to her) and sends it to Bob.
*   Bob uses this block to answer Alice.
*   **Benefit**: Bob places his answer in the network without knowing where Alice is. Alice receives it without exposing her IP to Bob or the network.

==== 3. Outbound Handles (The Mailbox)
An **Outbound Handle (OH)** is a specific Full Node where a Client has registered to receive messages.
*   It serves as the "Last Hop" of a Garlic chain.
*   It performs the **Push Notification** trigger.
*   It buffers messages while the client is sleeping.

==== 4. Two-Layer ACK System
To separate "Network Quality" from "User Behavior":
*   **Routing ACKs**: Generated automatically by the Outbound Node (e.g., "I received this package for Alice"). Used to score network path reliability.
*   **Channel ACKs**: Generated by Alice's Client (e.g., "I read the text"). Used for "Read Receipts".
