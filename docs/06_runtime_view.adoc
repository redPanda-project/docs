== Runtime View

This section details the core communication flows.

=== Scenario 1: Reverse Garlic (Stateless Reply Block)

To receive responses without leaking identity via DHT queries, Alice constructs a "Reverse Garlic" path.

==== 1. Build Phase (Alice)
Alice wants to enable Bob to reply to her.


1.  Alice selects a path of `n` hops: `H1 -> H2 -> ... -> Hn -> Alice(OH)`.
2.  Alice constructs the message layer-by-layer **from inside out**:
    *   **Inner Layer**: Final Delivery to her own Outbound Handle (OH).
    *   **Layer n**: Encrypted for `Hn`.
    *   ...
    *   **Layer 1**: Encrypted for `H1`.
3.  Result: A **`ReverseGarlicBlock` (RGB)**.

==== 2. Use Phase (Bob)
Bob wants to send the reply.


1.  Bob takes the `RGB` provided by Alice.
2.  Bob encrypts his payload with the session key defined in the RGB.
3.  Bob sends `(RGB, Payload)` to `H1` (the entry hop).

==== 3. Forward Phase (Network)
Each Hop `Hi`:


1.  Receives packet.
2.  Decrypts its layer.
3.  Finds `Next Hop` and `Next Key`.
4.  Forwards to `Hi+1`.
5.  *Stateless*: The hop retains no memory of the packet.

=== Scenario 2: ACK-Flows

We distinguish two types of acknowledgments to ensure reliable profiling.

==== Flow A: Routing-ACK (Network Reliability)
_Goal: Measure if the Outbound Node works._

1.  **Origin**: Generated by the **Outbound Node** (Last Hop) immediately upon receiving and validating the Garlic packet.
2.  **Path**: Sent back to the sender via a pre-crafted Reply Block (contained in the forward message).
3.  **Meaning**: "The network successfully delivered the packet to the destination mailbox."

==== Flow B: Channel-ACK (User Read Receipt)
_Goal: Confirm User has read/received the message._

1.  **Origin**: Generated by **Alice's Light Client** after decryption.
2.  **Path**: Sent as a new standard message from Alice to Bob.
3.  **Meaning**: "Alice has locally received and processed the message."
