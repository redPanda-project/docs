== Risks and Technical Debt

=== Technical Debt


*   **Signature vs Encryption Keys**: Currently, the protocol description uses ECDH keys partly for authentication. This needs to be cleanly separated into Signing Keys (Ed25519) and Encryption Keys (X25519) to avoid cryptographic confusion.
*   **RC4 Legacy**: Old references to RC4 must be removed from all implementations. Only AES/ChaCha is allowed.

=== Risks


*   **DHT Metadata Leakage**: Even with Reverse Garlic, the *lookup* of the Rendezvous Record (OHDescriptor) might leak interest if not carefully designed (e.g., if the Key is derived deterministically from the Public Key).
*   **Push Privacy**: Dependency on Google/Apple Push means we leak "User is receiving a message now" metadata to Big Tech, even if the content is hidden. Mitigation options (self-hosted UnifiedPush/ntfy, batching/random delay, polling-only mode for high-risk users) are a **decision for MS07**, not an accepted permanent state — see `threat_model.adoc`.
*   **No Forward Secrecy (static channel key)**: `K_enc` is a static, lifetime channel key (it even travels in the channel QR). A single key leak — device seizure, backup, screenshot of the QR — compromises the channel's **entire past and future**, and full nodes that stored ciphertext can decrypt it retroactively. This is unaddressed until **MS03b (Forward Secrecy)** introduces a symmetric ratchet and later an X25519 DH ratchet. Highest-severity cryptographic gap; cost grows the longer it is deferred.
*   **Reputation / Incentive Layer is Unsolved Research**: The full-node economy rests on MS09's reputation gossip with transitive trust weighting — a known, largely *unsolved* research problem (collusion, whitewashing, slandering; EigenTrust-class systems fail exactly here). MS09 should be treated as a **research phase**, not a deliverable milestone. Pragmatic fallback to ship in the meantime: a **curated bootstrap node list + local scoring** (the frontend already has `peer_stats`). Without working incentives the network degenerates to "the project team's few nodes" — de-facto centralized without a central operator's reliability.
*   **Unauthenticated Deposit / Mailbox Flush**: Anyone who knows an `oh_id` can deposit into that mailbox; with drop-oldest eviction a spammer can silently displace real messages (the `mailbox_overflow` flag only reports the loss). There is no per-connection register rate limit and no per-`MailItem` size limit. Unaddressed until **MS02b** (reject-new eviction, per-item size limit, byte quota, register rate limit — wiring the unused `RATE_LIMIT`/`QUOTA_EXCEEDED`/`BAD_REQUEST` status codes).
