== Risks and Technical Debt

=== Technical Debt


*   **Signature vs Encryption Keys** — *resolved (MS03, 2026-06-12)*: The protocol used ECDH keys partly for authentication. MS03 separated this cleanly into Ed25519 signing keys and X25519 encryption keys (node identities, OH auth, channel keys) — see the MS03 Decisions. Remaining legacy surface: the v22 handshake and the ECDSA fallback in `OutboundAuth` are still accepted for old clients; switching them off is a tracked follow-up (see `threat_model.adoc` (A)).
*   **RC4 Legacy** — *resolved*: No RC4/ArcFour references remain in either implementation (verified 2026-07-09). The v23 transport is AES-256-GCM since MS03; the only remaining non-GCM surface is the legacy v22 handshake (unauthenticated AES-CTR) behind `ACCEPT_LEGACY_V22_LIGHT_CLIENTS` — see the key-separation entry above and `threat_model.adoc` (A).

=== Risks


*   **DHT Metadata Leakage**: Even with Reverse Garlic, the *lookup* of the Rendezvous Record (OHDescriptor) might leak interest if not carefully designed (e.g., if the Key is derived deterministically from the Public Key).
*   **Push Privacy**: Dependency on Google/Apple Push means we leak "User is receiving a message now" metadata to Big Tech, even if the content is hidden. Mitigation options (self-hosted UnifiedPush/ntfy, batching/random delay, polling-only mode for high-risk users) are a **decision for MS07**, not an accepted permanent state — see `threat_model.adoc`.
*   **No Forward Secrecy (static channel key)** — *mitigated (MS03b, 2026-06-12)*: `K_enc` was a static, lifetime channel key; a single key leak compromised the channel's entire past and future, and full nodes that stored ciphertext could decrypt it retroactively. MS03b introduced a Double Ratchet (Envelope v4, combined symmetric + X25519 DH ratchet): after a channel's first full round trip, message keys contain fresh DH entropy that `K_enc` cannot reproduce, and old keys are not retained (best-effort deletion — Dart cannot zeroize GC memory). Residual exposure — current ratchet state, bounded skipped keys (512/1024/30 days), channels before their first round trip, and `K_enc` still travelling in the QR — is documented in `threat_model.adoc` (E).
*   **Reputation / Incentive Layer is Unsolved Research**: The full-node economy rests on MS09's reputation gossip with transitive trust weighting — a known, largely *unsolved* research problem (collusion, whitewashing, slandering; EigenTrust-class systems fail exactly here). MS09 should be treated as a **research phase**, not a deliverable milestone. Pragmatic fallback to ship in the meantime: a **curated bootstrap node list + local scoring** (since MS06 the frontend maintains R-ACK-based node scores and prefers higher-scored hops). Without working incentives the network degenerates to "the project team's few nodes" — de-facto centralized without a central operator's reliability.
*   **Unauthenticated Deposit / Mailbox Flush** — *mitigated (MS02b, 2026-06-11)*: Anyone who knew an `oh_id` could deposit into that mailbox and, with drop-oldest eviction, silently displace real messages. MS02b switched to reject-new eviction and added a per-`MailItem` size limit, a per-mailbox byte quota, and a register rate limit (wiring the previously unused `RATE_LIMIT`/`QUOTA_EXCEEDED`/`BAD_REQUEST` status codes) — see the MS02b Decisions. Deposits remain unauthenticated by design (senders are anonymous); authenticated announces and a deposit rate limit per source were deliberately deferred out of MS02b.
